# 老板说分析一下

## 目标

把一句模糊的“分析一下”接成一份可交付的判断：先接收需求和判断物料，再确认分析方法，最后输出能帮助老板、客户或团队继续决策的分析结果。

## 核心原则

- 先接住需求，再判断是否需要反问；反问只问会影响分析方向的关键问题。
- 主动补充并判断可分析物料：能联网查的就建议联网查，需要用户补的就明确说明。
- 用户表示不能提供更多物料时，不要拒绝分析；继续基于现有材料给出低/中/高可信度判断。
- 在正式分析前，先简述将使用的分析方法、模型和原因，等待用户确认后再继续；如果用户明确要求直接分析，可以跳过等待。
- 先分析数据、表象和现状，再分析具体问题、原因、影响和动作。
- 最终输出固定为“表象分析、具体分析、给领导可以这么说”。
- 删除漂亮废话；每个结论都要能回答“所以现在该怎么办”。

## 工作流

### 1. 接收需求

先把用户的一句话需求转成可分析任务。判断分析对象和分析目标，必要时反问。

分析对象可包括：

- 内容：文章、视频、账号、标题、选题、传播效果。
- 竞品：产品动作、内容策略、商业动作、活动打法。
- 数据：增长、下滑、转化、留存、异常波动。
- 新闻/趋势：行业事件、AI 产品发布、政策变化、平台规则变化。
- 产品/功能：工具、功能、体验、需求、替代方案。
- 方案/机会：某件事值不值得做、是否跟进、如何投入。

分析目标可包括：

- 解释原因：为什么变好、变差、变热或变冷。
- 判断价值：值不值得跟进、复用、投入或放弃。
- 比较优劣：A 和 B 哪个更适合当前目标。
- 识别风险：机会背后有什么坑。
- 预测影响：对我们、客户、用户、行业有什么影响。
- 给出行动：下一步做什么、观察什么、暂时不做什么。
- 提炼经验：哪些方法可以迁移到我们自己身上。

如果需求仍然模糊，先问 1-3 个关键问题，不要一次性列太多。例如：

```text
我先确认一下：这次分析主要是想解释原因、判断值不值得跟进，还是给下一步动作？
如果你没有更多背景，我也可以先按“给老板判断下一步怎么处理”的目标继续。
```

如果用户没有回复或表示不用补充，基于最可能的目标继续。

### 2. 补充并判断可分析的物料

梳理已有物料，再判断还可以补什么。补充物料有两种方式：

1. 联网补充公开信息。
2. 要求用户提供其他数据或物料。

不要把缺材料说成无法分析。用户表示不能提供时，继续分析，但标注可信度和不确定项。

```text
已有材料：
- 用户已经提供的链接、截图、数据、描述、文档或背景。

可以联网补充的公开信息：
- 官方公告、产品文档、官网、新闻稿。
- 权威媒体、行业报告、公开数据、研究论文。
- 竞品官网、社媒账号、内容样本、招聘信息、应用商店信息。
- 用户公开评论、社区讨论、平台榜单、搜索趋势。

需要用户提供的非公开信息：
- 公司内部业务数据、历史同期数据、渠道拆分、转化漏斗。
- 销售、客服、运营反馈和用户访谈。
- 投放、活动、价格、库存、资源位、渠道、人群变化。
- 老板真正关心的判断口径、预算约束、时间节点和组织偏好。
```

材料不足时，按以下方式继续：

```text
材料判断：
目前可以先做初步分析，但结论可信度为低/中/高。

建议补充：
1. 联网信息：建议查……
2. 用户补充：建议补……
3. 当前可先判断：……
4. 暂时不能确认：……
```

如果任务明显依赖最新公开信息，先建议联网补充；如果用户要求不联网或当前不能联网，继续用已有材料分析并说明限制。

### 3. 确认分析方法

正式分析前，先向用户简述计划使用的方法、模型和原因，并等待确认。

输出要短，像这样：

```text
我建议这样分析：
1. 先做表象分析：看趋势、结构和异常点，判断问题掉在哪一层。
2. 再做具体分析：用漏斗分析 + 对照分析，判断最可能原因和验证动作。
3. 最后输出一段可以直接跟领导说的话。

你确认的话，我就按这个方法继续。
```

如果用户已经明确说“直接分析”“不用确认”“继续”，可直接进入下一步。

方法选择规则：先写定制化拆解思路，再选择 0-2 个专业模型辅助。只有当模型能提升判断质量时才使用。

常见选择：

- 内容/竞品：选题、痛点、结构、传播机制、可复制性；可用 AIDA、用户旅程、Benchmarking、差异化分析。
- 数据异常：趋势、漏斗、渠道、变量、假设排序；可用漏斗分析、5 Whys、MECE、归因分析、对照分析。
- 行业趋势：事实、相关性、影响路径、时间尺度、机会风险；可用 PEST、波特五力、价值链、情景分析。
- 产品/功能：用户场景、能力边界、替代关系、使用成本、落地阻力；可用 JTBD、KANO、RICE、SWOT、用户旅程。
- 方案/机会：目标、收益、成本、风险、资源匹配、验证路径；可用 ROI、ICE、RICE、风险矩阵、决策树。

需要更细的模型选择时，读取 `references/analysis-models.md`。

### 4. 进行分析

先做表象分析，再做具体分析。

#### 4.1 表象分析

先描述看得见的现状，不急着下根因结论：

- 数据趋势：变好、变差、持平，是否超过正常波动。
- 结构变化：渠道、人群、产品、地区、时间段是否变化。
- 漏斗断点：曝光、点击、到达、留资、成交、复购哪一层异常。
- 外部表象：舆论、评论、平台规则、竞品动作、行业事件是否有变化。
- 事实边界：哪些是已有材料能支持的，哪些还只是现象。

#### 4.2 具体分析

再围绕问题建立假设并验证。至少提出 2-3 个可验证假设，而不是只罗列信息。

```text
H1：这次变化主要来自……
H2：这次变化主要来自……
H3：这可能只是短期噪音/局部案例/表层现象。
```

逐条检查：

- 支持证据是什么。
- 反驳证据是什么。
- 还缺什么材料。
- 当前最可能的解释是什么。

必要时用 5 Whys 追根因，但每一层都要标注证据或验证方式，不要把猜测写成事实。

### 5. 输出结果

最终输出固定为三段：

```text
1. 表象分析
- 先说目前看到了什么变化。
- 说明变化发生在哪些指标、环节、对象或时间点。
- 标注当前材料可信度和材料缺口。

2. 具体分析
- 给出最可能原因。
- 说明支持证据、反驳证据和还需要验证的点。
- 给出建议动作：现在做什么、继续观察什么、暂时不要做什么。

3. 给领导可以这么说
- 用 3-6 句话，写成可以直接转述给老板/领导的口径。
- 先给结论，再给依据，再给下一步动作。
- 不要写过程解释和模型名。
```

## 自检去空话

交付前检查：

- 是否按“表象分析、具体分析、给领导可以这么说”输出。
- 是否说清楚这次分析服务什么决策。
- 是否区分事实、推测、判断和建议。
- 是否指出联网可补充信息和需要用户提供的物料。
- 是否在分析前确认过分析方法，或用户已明确要求直接继续。
- 是否先讲表象，再讲具体问题。
- 是否给出下一步动作和不确定项。
- 是否删掉“具有重要意义”“值得关注”“未来可期”等无判断密度的表达。
